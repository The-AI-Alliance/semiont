# Streaming Architecture

## How async operations work

Every async operation in the codebase follows one pattern:

```
Client                              Server
  │                                   │
  │  POST /resources/:id/...         │
  │  { body }                         │
  │ ──────────────────────────────►   │
  │                                   │  create job / emit command
  │  ◄── 202 { correlationId }       │
  │                                   │
  │                                   │  actor processes...
  │                                   │  worker emits progress events
  │                                   │  on scoped EventBus
  │                                   │
  │  events-stream (long-lived SSE)  │
  │  ◄── event: mark:progress        │
  │  ◄── event: yield:progress       │
  │  ◄── event: mark:body-updated    │
  │  ◄── event: mark:assist-finished │
  │                                   │
```

**Commands** are plain HTTP POST requests. They return `{correlationId}` immediately. The server-side actor or job worker processes the command asynchronously.

**Results** arrive on the **events-stream** — a single long-lived SSE connection per resource that every client viewing that resource subscribes to. Progress events, completion events, and state-change events all flow through the same stream.

## The events-stream

`GET /resources/:id/events/stream`

A long-lived SSE connection opened once per resource view. Carries two categories of events:

### Persisted events (from `EventStore.appendEvent`)

Domain events written to the event log: `mark:added`, `mark:body-updated`, `yield:created`, `job:completed`, etc. These have `metadata` (sequenceNumber, checksum) and are delivered as `EnrichedResourceEvent` — the `StoredEvent` extended with optional post-materialization fields (e.g., the full `annotation` for annotation-mutation events).

The subscription list is derived at compile time from `PERSISTED_EVENT_TYPES` in `@semiont/core`. Adding a new persisted event type without updating the list fails the build.

### Command-result events (from actors on the scoped EventBus)

Ephemeral progress and result events from actors: `mark:progress`, `yield:progress`, `yield:finished`, `yield:failed`, `match:search-results`, `gather:complete`, `mark:assist-finished`, etc. These don't have `metadata` and are delivered as raw JSON payloads.

The subscription list is `STREAM_COMMAND_RESULT_TYPES` in `@semiont/core`.

### Enrichment

For events that mutate an annotation (`mark:added`, `mark:body-updated`, `mark:removed`), the events-stream reads the post-materialization annotation from the view storage and includes it on the SSE payload. This lets the frontend's `semiont.browse` namespace update its cached Observables in-place without an HTTP refetch. The enrichment logic lives in `event-stream-enrichment.ts`.

### Reconnection + replay

The events-stream supports the W3C SSE `Last-Event-ID` replay mechanism. Each SSE event's `id` is set to the event's `sequenceNumber`. On reconnect, the client sends its last seen id via the `Last-Event-ID` header, and the server replays missed events from the event log.

A replay cap (`REPLAY_WINDOW_CAP = 1000`) prevents unbounded replays. If a client has been disconnected too long, the server sends a `replay-window-exceeded` event and the client does a cold refetch.

The `createSSEStream` factory in `@semiont/api-client` handles reconnection automatically with exponential backoff (1s → 30s cap) when `reconnect: true` is set. The three long-lived stream methods (`resourceEvents`, `globalEvents`, `attentionStream`) all have this enabled.

## correlationId

Every command carries a `correlationId` (UUID generated by the client or the route handler). The id is threaded through:

1. The POST request body (or generated by the route)
2. The EventBus command event
3. `EventStore.appendEvent` metadata (for persisted events)
4. The actor's result/progress events

Clients use correlationId to match results back to their own pending operations. Other participants see the same events but with someone else's correlationId — they still see the state change but don't match it to a local pending operation.

## No per-operation SSE streams

Previously, each async operation had its own bespoke SSE route (`bind-annotation-stream`, `match-search-stream`, `gather-annotation-stream`, `yield-resource-stream`, `annotate-references-stream`, etc.). These have been deleted. All results flow through the single events-stream.

The per-operation routes were a source of fragility: the UI's correctness depended on both the per-operation stream and the events-stream being healthy simultaneously. The unified architecture eliminates that coupling — one stream, one delivery path.

## Frontend consumption

The `BrowseNamespace` in `@semiont/api-client` (`semiont.browse`) subscribes to the EventBus (which the SSE auto-router populates from the events-stream). On `mark:body-updated`, it reads the enriched annotation and updates the cached Observable in-place — no HTTP refetch.

Verb namespaces orchestrate commands: `semiont.bind.body()` calls the HTTP POST; `semiont.mark.assist()` calls the POST and returns an Observable that filters EventBus progress events by correlationId. The events-stream delivers results to all participants.

React hooks (`useResourceEvents`, `useMarkFlow`, `useYieldFlow`, `useContextGatherFlow`) bridge EventBus UI events to namespace method calls and update local React state from EventBus domain events.

## Adding a new async operation

1. Add the request/response OpenAPI schemas
2. Create a POST route handler that emits a command on the EventBus and returns `{correlationId}`
3. Implement the actor/worker handler that processes the command and publishes results on `eventBus.scope(resourceId)`
4. Add the result event type to `STREAM_COMMAND_RESULT_TYPES` (for non-persisted events) or `PERSISTED_EVENT_TYPES` (for persisted events)
5. Add a method to the appropriate verb namespace in `@semiont/api-client` (e.g., `semiont.mark.assist()` for a new mark operation)

No new SSE route. No new SSEClient method. No streamSSE handshake. No keep-alive ping. No abort handler.
